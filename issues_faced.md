# Known Issues & Resolutions Log

This document tracks significant bugs, data isolation issues, and critical system failures encountered during development and testing, along with their root causes and resolutions.

## Issue 1: RAG History Data Isolation Failure
**Phase:** Beta Testing & Hardening / RAG Feature Implementation  
**Date Identified:** March 3, 2026  
**Severity:** CRITICAL (Data Privacy Leak)

### Description
A new user account with zero saved trips or history was able to query the RAG generic history endpoint and receive answers containing trip details generated by another user (e.g., "10 Downing St to Buckingham Palace"). 

### Root Cause Analysis
The issue was caused by a compound failure of state management between the SQLite database, ChromaDB vector store, and Python async context variables:
1. **De-synchronized Vector Storage:** During iterative testing, the SQL database was periodically wiped or reset, but the persistent ChromaDB collections were not. When `.query()` was called on an "empty" user collection, it retrieved stale orphaned vectors that no longer had associated SQL records.
2. **Context Leakage Across Async Requests:** In `backend/app/agent/core.py`, the `contextvars` used to scope `user_id_ctx` were only updated `if user_id is not None`. Because the LangChain AgentExecutor runs within a shared `asyncio` event loop, a request missing a user_id would accidentally inherit the stale `user_id_ctx` from a completely different concurrent or past request.
3. **Falsy Check Bug:** In `backend/app/agent/tools.py`, the check `if not user_id:` treated `0` (used for the demo endpoint) as falsy, triggering short-circuit logic incorrectly.

### Resolution Implemented
1. **SQL as Source of Truth Check:** Added a strict check inside `rag_service.py` to first verify if the user has any trips or history *in SQL*. If SQL is empty but ChromaDB returns documents, the pipeline now actively **purges the stale vector data** and returns a generic "No trips yet" response.
2. **Startup ChromaDB Sync:** Built `chroma_sync.py` and hooked it into the FastAPI `startup` event. On boot, the server automatically scans and purges all ChromaDB collections that belong to deleted users or lack corresponding SQL records.
3. **Context Isolation Lock:** Enforced explicit unconditional resets of `user_id_ctx` and `user_city_ctx` at the start of every LangChain agent invocation, ensuring airtight context isolation per request.
4. **Tool Falsy Fix:** Replaced all `if not user_id:` checks with `if user_id is None:` across all vector agent tools.
5. **Purged Legacy Collections:** Deleted archaic, non-namespaced collections (`trip_history`, `saved_trips`, `saved_stops`) from an earlier architecture phase to eliminate residual risk.

## Issue 2: User City Not Refreshing After Profile Update
**Phase:** Core Agent Interaction / User Profile Polish  
**Date Identified:** March 3, 2026  
**Severity:** MEDIUM (Degraded UX)

### Description
When a user updated their city/state via the Account Screen, the intelligent agent continued to use their old location when geocoding implicitly relative queries (like "home" or "main street").

### Root Cause Analysis
In `backend/app/routers/agent.py`, the `user_city` was fetched using `current_user.profile.full_location`. Because the `current_user` object and its `.profile` relationship were often cached in the active SQLAlchemy session during the request lifecycle (from auth middleware), the location data would go stale and not reflect changes made via the `UpdateProfileRequest` endpoint until the user fully logged out and back in.

### Resolution Implemented
1. **Fresh Database Queries:** Modified the chat endpoint in `agent.py` to bypass the cached `.profile` attribute. It now actively queries the `models.UserProfile` table directly on every agent invocation using the current user's ID. This guarantees the geocoding context always matches the most recently updated location in the database.

## Issue 3: Agent Halting Early on Multi-stop Routes
**Phase:** Agent Optimization / Geocoding Workflow  
**Date Identified:** March 3, 2026  
**Severity:** HIGH (Core Functionality Failure)

### Description
When a driver pasted a large batch of stops (e.g., 6+ stops) into the chat, the agent would only process the first 1-2 stops, prematurely conclude its ReAct loops, and reply with "Safe driving!" without mapping the rest of the list.

### Root Cause Analysis
Large LLMs naturally gravitate towards brevity to conserve output tokens. The agent's system prompt told it to "GEOCODE ALL OF THEM AT ONCE" but lacked a strict procedural directive. When the agent saw a long list, it would get computationally "lazy," deciding it had accomplished enough after a loop or two and jump straight to the Final Answer out of convenience.

### Resolution Implemented
1. **Procedural "Counting" Directive:** Updated the Geocode Batching instructions in `prompts.py` to exploit LLM behavior with a strict counting algorithm. The prompt now mandates: 
   *"Do NOT stop early! First, count how many stops the user gave you. Then, keep taking Action: geocode_stop for each one sequentially."*
   Forcing the LLM to internally "count" the objective items anchors its attention and drastically reduces the chance of premature halting until its internal loop satisfies the exact count.

## Issue 4: 503 Capacity Errors Bypassing Key/Model Rotation
**Phase:** LLM Orchestration / Error Handling
**Date Identified:** March 4, 2026
**Severity:** HIGH (Unnecessary User-Facing Errors)

### Description
Even though a robust multi-tiered key and model rotation system was built, internal `503 Service Unavailable` or capacity errors from Groq/Gemini were bubbling up to the user as a generic "Something went wrong" message. The system was failing to silently catch these capacity issues and rotate.

### Root Cause Analysis
The catch block in `backend/app/agent/core.py` was dependent on the `_is_rate_limit_error()` helper function in `groq_client.py`. This helper was strictly looking for `RateLimitError` class names and `429` status tokens. It completely ignored `InternalServerError`, `APIStatusError`, `503`, and `capacity` keywords, which are often thrown by API providers when their inference clusters are overloaded. Because it returned `False` for a 503 error, the system bypassed the rotation fallback flow and hard-crashed the request back to the frontend.

### Resolution Implemented
1. **Expanded Fallback Trigger Tokens:** Updated `_is_rate_limit_error()` to actively match `InternalServerError`, `APIStatusError`, `ServiceUnavailableError`, as well as string tokens like `503`, `500`, `502`, `504`, `capacity`, and `overloaded`. Now, any server-side overload from the LLM provider will seamlessly trigger the existing model, key, and provider rotation flows without exposing the error to the driver.

## Issue 5: Famous National Landmarks Failing to Geocode
**Phase:** Core Agent Interaction / Geocoding Workflow
**Date Identified:** March 4, 2026
**Severity:** HIGH (Core Functionality Failure)

### Description
Famous national landmarks like "Times Square" or "Central Park" were failing to resolve with "I couldn't find this stop" if they were routed through the tertiary national-level fallback.

### Root Cause Analysis
Two overlapping issues were breaking the national cascade:
1. **Hardcoded State Component Bias:** The Google API `components` parameter inside `_fetch_geocode` was hardcoded to `"administrative_area:NY|country:US"`, meaning searches from users in other states were inadvertently being trapped within New York constraints during earlier cascade attempts.
2. **Aggressive Token Filtering:** The tertiary "national" fallback attempt (Attempt 3) was subject to `_filter_fallback_results`, which strictly stripped out any Google results if their `formatted_address` didn't explicitly contain the driver's exact input words. For "Times Square," Google natively returns `"Manhattan, NY 10036"`. Since this string doesn't contain the word "Times", the fallback function threw the valid result in the trash.

### Resolution Implemented
1. **Dynamic Component Binding:** Adjusted the `components` API payload to intelligently dynamically bind to the user's `default_state` instead of hardcoding `NY`.
2. **Unrestricted National Base:** Completely ripped the `_filter_fallback_results` logic out of Attempt 3. National fallback queries now pass `use_bias=False` straight to Google and implicitly trust whatever Google's global prominence algorithm returns first, correctly matching landmarks even when their colloquial names differ from their formatted mailing addresses. 
3. **Agent Prompt Unshackled:** Re-wrote the system prompt missing-stop rule, scrubbing mentions of "in your area" from the examples, and adding a `National Routing` directive to ensure the LLM knows it has permission to route famous landmarks anywhere in the country.

## Issue 6: Improper National Fallback Causing Out-of-State Hallucinations
**Phase:** Core Agent Interaction / Geocoding Logic
**Date Identified:** March 4, 2026
**Severity:** MEDIUM (Bad User Experience)

### Description
Famous national landmarks geocoding was returning the wrong coordinates from the user's base state if the national fallback caught them without confirmation context.

### Root Cause Analysis
Attempting to catch generic national queries in `geocoding_service.py` with an unrestrained `Attempt 3: National Fallback` without first validating the driver's intent resulted in local stops occasionally being misinterpreted as far-away national matches, while legitimate out-of-state routing queries returned chaotic results based on the driver's registered profile location.

### Resolution Implemented
1. **Removed National Fallback:** Completely deleted `Attempt 3` from `geocoding_service.py`. The system now strictly respects the user's state constraints within the backend API wrapper to prevent false positives.
2. **LLM `override_city` Context Control:** Updated the `geocode_stop` tool signature in LangChain to cleanly expose an `override_city` argument.
3. **Out-of-State Agent Interrogation Flow:** Rewrote the LLM System Prompt. If the AI detects a user requesting stops distinctly outside their registered state, it must pause and interrogate the driver *once*: *"It looks like these stops are in [DETECTED STATE, e.g. New York]. Are you currently in [DETECTED STATE]?"* The prompt explicitly forbids the agent from accidentally asking if the user is in their home state ("Texas") during this flow. Once explicitly confirmed, the LLM actively passes the target state into the `override_city` tool parameter for the duration of the conversation, effectively hot-swapping the user's geocoding biases dynamically.

## Issue 7: Agent Hallucinating Coordinates Due to ReAct JSON Parsing Bug
**Phase:** Core Agent Interaction / Geocoding Workflow
**Date Identified:** March 4, 2026
**Severity:** MEDIUM (Frontend Map Breaks / Missing Route List)

### Description
The LLM accurately deduced it needed to execute out-of-state routing by passing an `override_city` argument to the `geocode_stop` tool. However, the tool failed to geocode the stop, causing the LLM to get frustrated, bypass the tool entirely, and hallucinate raw coordinates directly inside its `Final Answer` text. Because it bypassed the tool, the structural `stops` array was dropped, meaning the frontend chat couldn't display the "Preview Route" button, and the Map screen received fallback default coordinates from Texas instead of the hallucinated NYC ones.

### Root Cause Analysis
LangChain's zero-shot ReAct agent format has a known limitation when passing multi-parameter arguments into its tools. The string parser is fragile. When the LLM attempted to pass two variables to the tool, it collapsed them both into a literal JSON dictionary string injected solely into the first argument:
`query = '{"query": "Times Square", "override_city": "New York, NY"}'`
Because this dictionary string physically occupied the first argument space, the `geocode_stop_tool`'s actual `override_city` kwarg was left as its default empty `""` value. It then helplessly sent that exact raw JSON dict string to the Google Maps API, which immediately returned "Could not find this stop".

### Resolution Implemented
1. **JSON Interception Hook:** Modified `geocode_stop_tool` in `backend/app/agent/tools.py` with defensive JSON-parsing guardrails. 
2. **Payload Extraction:** When the tool receives the `query` argument, it actively checks if the string opens and closes with raw curly braces `{ }`. If so, it invokes `json.loads(query)`, safely splits the dictionary back into `parsed_query` and `parsed_override` variables, and then overwrites the dummy `""` parameter correctly before pushing the cleanly extracted arguments to `geocoding_service.geocode()`.

## Issue 8: Stale Coordinates Persisting After Out-of-State Re-Geocoding
**Phase:** Core Agent Interaction / Stop Assembly
**Date Identified:** March 4, 2026
**Severity:** HIGH (Map and Route URL Point to Wrong Location)

### Description
When the agent geocodes a stop twice (first attempt â†’ home state coords, second attempt with `override_city` â†’ correct out-of-state coords), the Leaflet map and Google Maps URL still used the wrong first-attempt coordinates. This is fully generic â€” it affected any home-state / target-state combination, not just Dallas/NYC.

### Root Cause Analysis
`_extract_stops_from_steps()` in `backend/app/agent/core.py` blindly appended **every** successful `geocode_stop` result into a flat list. When the same stop was geocoded twice (before and after out-of-state confirmation), both results appeared as separate entries â€” producing 4 stops instead of 2. The comparison logic at line 252 then picked the longer `stops_tools` list (4 items with stale coords) over the correct `stops_reply` list (2 items from Final Answer text). Additionally, the `label` field for JSON-input tool calls stored the raw JSON string instead of the clean query name.

### Resolution Implemented
1. **Label-Keyed Deduplication:** Replaced the flat `stops` list with an ordered `stops_by_label` dictionary keyed by the normalized (lowercased) stop label. When the same stop is geocoded again, the new result **overwrites** the old entry, guaranteeing only the last successful coordinates survive.
2. **JSON Label Cleanup:** Added parsing logic to extract clean labels from LangChain's JSON dict string inputs (e.g. `{"query": "Times Square", "override_city": "New York, NY"}` â†’ `"Times Square"`).
3. **Position Rebuild:** Positions are recalculated after merging trip stops and geocoded stops, ensuring a clean sequential order.

## Issue 9: LangChain ReAct Silently Dropping override_city Parameter
**Phase:** Core Agent Interaction / Geocoding Workflow
**Date Identified:** March 4, 2026
**Severity:** CRITICAL (All Out-of-State Routing Completely Broken)

### Description
After confirming "Yes" to out-of-state routing, the agent attempted to pass `override_city` as a second `Action Input (override_city):` line. LangChain's ReAct parser **only reads the first `Action Input:` line** and silently drops everything else. The tool received `query="Times Square"` with `override_city=""`, geocoding defaulted to the user's home city (Dallas), and every attempt returned Dallas coordinates. The LLM retried 7+ times in a frustrated loop before timing out.

### Root Cause Analysis
LangChain's zero-shot ReAct format is fundamentally single-parameter per tool action. The format is strictly:
```
Action: tool_name
Action Input: single_string_here
```
There is no native support for multiple named parameters. Our `override_city` parameter was invisible to the parser regardless of how the LLM formatted it (separate line, JSON dict, etc.). The tool always received `override_city=""`.

### Resolution Implemented
1. **Eliminated override_city Parameter:** Removed the `override_city` parameter from `geocode_stop_tool` entirely. The tool is now back to a clean single-parameter `query: str` signature.
2. **Query-Appended State:** Rewrote the system prompt to instruct the LLM to append the confirmed state directly into the query string: `Action Input: Times Square, New York, NY` instead of trying to pass a separate parameter. This works perfectly within ReAct's single-input constraint.
3. **JSON Safety Net:** Kept a defensive JSON parser in the tool as a fallback â€” if any model still tries to pass `{"query": "...", "override_city": "..."}` as a JSON dict, the tool will extract both fields and concatenate them into a single query string.

## Issue 10: Geocoding Service Ignoring State in Query, Always Using Home State Bias
**Phase:** Core Geocoding Engine
**Date Identified:** March 4, 2026
**Severity:** CRITICAL (All Out-of-State Routing Broken at API Level)

### Description
Even after fixing the LLM prompt and tool to correctly pass `"Times Square, New York, NY"` as the query, the geocoding service still returned Dallas, TX coordinates (32.77, -96.80). The Google API was being forced to search within Texas regardless of what the query string contained.

### Root Cause Analysis
Three compounding bugs in `geocoding_service.py`:
1. **Home-state appended to out-of-state query:** Attempt 1 blindly appended `user_city` (Dallas, TX) to the query, producing `"Times Square, New York, NY, Dallas, TX"`. Google saw "TX" and returned Dallas.
2. **`components` filter locked to home state:** The Google API `components` parameter was always set to `"administrative_area:{home_state}|country:US"` (TX), forcing Google to only return results within Texas.
3. **`_is_invalid_generic` rejecting NYC results:** The filter contained `if ", New York," in fmt_addr: return True` â€” which rejected ANY valid New York result as "generic".

### Resolution Implemented
1. **State Detection Engine:** Added `_extract_state_from_query()` with a full set of all 50 US state abbreviations + DC. When the query already contains a state (e.g. `"Times Square, New York, NY"`), the service automatically detects `NY` and uses it for all Google API biasing instead of the user's home state.
2. **Dynamic Bias Switching:** When a query state is detected, the `components` filter switches to that state, the `location` center is looked up for that state, and the query is sent to Google as-is without appending the user's home city.
3. **Removed New York Filter:** Deleted the `", New York,"` check from `_is_invalid_generic` since it was incorrectly rejecting valid NYC landmarks.

## Issue 15: 401 Unauthorized Loop Due to Supabase Backend Validation
**Phase:** Core Authentication / API
**Date Identified:** March 6, 2026
**Severity:** CRITICAL (Login Loop / App Unusable)

### Description
Immediately after successfully authenticating via the frontend, users would load the Home Screen for 1-2 seconds before being inexplicably booted back to the login screen. The frontend correctly possessed a valid Supabase JWT and added it to the `Authorization: Bearer <token>` header, but the backend rejected it with a `401 Unauthorized` response on endpoints like `GET /api/v1/auth/me`.

### Root Cause Analysis
The frontend correctly intercepted and hydrated the session. However, on the backend (`app/auth.py`), the `get_current_user` dependency invoked `supabase.auth.get_user(token)`. This command fundamentally triggers an outbound HTTP POST request from the FastAPI backend to the remote Supabase GoTrue Auth service server in order to validate the JWT. The Supabase server unexpectedly rejected this server-to-server proxy validation call, throwing an internal 401 to the backend. The backend then surfaced a fast `HTTP 401` back to the frontend. The frontend's Axios interceptor rightly caught this 401 and aggressively redirected the user back to `/login`, causing an infinite auth loop.

### Resolution Implemented
1. **Local JWT Verification:** Completely stripped the `supabase.auth.get_user(token)` remote network call from the fast-path authentication dependency.
2. **Zero-Latency Decoding:** Switched to decoding the JWT entirely locally on the backend server using the `python-jose` library (`jwt.decode()`).
3. **Injected JWT Secret:** Used `SUPABASE_JWT_SECRET` pulled from environment variables (`app/config.py`) and standard `HS256` symmetric signing to securely parse and implicitly validate the JWT payload (`sub` and `email`), converting it into a local `AuthUser` class replica without incurring network latency or remote provider rejection.

## Issue 16: Initial API Call Failing Due to Asynchronous Auth Hydration
**Phase:** Core Authentication / Frontend State
**Date Identified:** March 6, 2026
**Severity:** HIGH (Intermittent Login Loops / Flash of Unauthenticated Screen)

### Description
Immediately after the user logged in, the application redirected to the Home screen but the initial batch of API calls (`/trips`, `/history`) occasionally returned `401 Unauthorized`, kicking the user back out. This happened despite the credentials being perfectly valid and the backend 401 bug (Issue 15) being resolved.

### Root Cause Analysis
A race condition existed between Supabase JS's asynchronous local storage writes and the React tree mounting the `HomeScreen`. When `login()` succeeded, it called `supabase.auth.setSession()`, which writes to local storage asynchronously. However, the frontend `AuthScreen` immediately called `navigate('/home')` before this write finished. The `HomeScreen` instantly fired outbound Axios requests. The Axios request interceptor called `supabase.auth.getSession()` to grab the token, but because the storage write was actively pending, it returned `null`. The requests fired lacking the `Authorization: Bearer` header, which the backend rightfully rejected. 

### Resolution Implemented
1. **Interceptor Breather Wait-State:** Updated the Axios request interceptor in `api/client.js`. If the interceptor finds `session?.access_token` to be null on the first attempt, it intentionally forces a `500ms` asynchronous wait (`await new Promise(r => setTimeout(r, 500))`) before retrying the `getSession()` call. This allows pending Supabase storage writes to safely resolve before dispatching the outbound HTTP packet.
2. **Explicit Token Passing via Zustand:** Refactored `authStore.js` and `AuthScreen.jsx` so that the `login()` and `signup()` API responses directly return the extracted raw `access_token` and pass it synchronously into the Zustand `setUser(data.user, data.access_token)` mapper immediately prior to navigation, eliminating the frontend's reliance on asynchronous `supabase.auth.getSession()` for core React hydration states.

## Issue 17: White Screen on Navigation to Auth Routes (SPA Hard Reload)
**Phase:** Frontend Routing / Production Polish
**Date Identified:** March 7, 2026
**Severity:** MEDIUM (Intermittent White Screen UX Issue)

### Description
When navigating from the landing page to the Login or Signup screen, the app occasionally rendered a completely blank white screen. A manual browser refresh was required to make the page finish loading.

### Root Cause Analysis
The LandingPage CTA buttons, Desktop Navbar links, and Mobile Menu items were built using standard HTML `<a>` tags (e.g., `<a href="/login">`). Since RoutAura is a Single Page Application (SPA), clicking a standard anchor tag forces the browser to discard the current DOM and perform a full "hard reload" of the page. This hard reload triggered during routing caused a race condition with the authentication state's core hydration logic (which runs on the initial component mount cycle), occasionally resulting in a blank white screen render if the hydration check evaluated out of sync with the React Router.

### Resolution Implemented
1. **Client-Side Navigation:** Replaced all `<a>` tags pointing to internal routes throughout the `LandingPage.jsx` (Hero CTAs, Navbar, Mobile Menu) with React Router `<Link to="...">` components.
2. **Hydration Protection:** This ensures navigation happens instantly client-side via React Router without destroying the DOM or triggering full browser hard-reloads, completely bypassing the hydration race condition that caused the white screen.

## Issue 18: Profile Data Missing from Account Screen After Fresh Login
**Phase:** Core Authentication / Profile Data
**Date Identified:** March 7, 2026
**Severity:** MEDIUM (Confusing UX: "Data loss" illusion)

### Description
When a user updated their personal information (First Name, Last Name, Birthday) and saved it, the data persisted correctly. However, if they signed out and signed back in, the Account screen would inexplicably show blank fields for those values. A hard refresh of the page (F5) would magically bring the data back.

### Root Cause Analysis
The frontend uses Zustand to manage the globally active `User` object. 
1. When a user did a hard-refresh, the `AppShell` triggered a `hydrate()` loop which called the `GET /api/v1/auth/me` endpoint. This endpoint correctly queried the `UserProfile` table and returned all populated fields (`first_name`, `last_name`, `birthday`, `city`, `state`, etc.). 
2. But when a user actively submitted the Login form, the `AuthScreen` directly updated Zustand using the exact JSON dictionary returned by the `POST /api/v1/auth/login` endpoint.
3. The `POST /api/v1/auth/login` backend endpoint was only programmed to return `id`, `email`, `city`, `state`, and `zip_code`. It entirely omitted `first_name`, `last_name`, and `birthday` in its payload. 
Thus, logging in actively overwrote the frontend's Zustand state with a `User` object missing those three fields, giving the UI the illusion that the data was erased.

### Resolution Implemented
1. **Endpoint Parity:** Updated the `/login` route in `backend/app/routers/auth.py` to faithfully extract and include `first_name`, `last_name`, and `birthday` from the `UserProfile` database row into the returned `user` dictionary.
2. **Signup Parity:** Also added the missing `birthday` field to the payload returned by the `/signup` route.
Now, the `User` object passed to the frontend is perfectly identical whether it comes from a fresh Login, a Signup, or a background Hydration (`auth/me`), ensuring the Account screen always has the complete data immediately.

## Issue 19: "Cannot Validate Credentials" on All API Calls (Production)
**Phase:** Core Authentication / Production
**Date Identified:** March 7, 2026
**Severity:** CRITICAL (App Completely Unusable)

### Description
After deploying to production (Railway), every authenticated API call â€” including chat, trips, and history â€” returned "Could not validate credentials." The frontend's Axios interceptor correctly attached the `Bearer` token, but the backend rejected it.

## Issue 36: Identical Messages Blocked Across Different Sessions (Duplicate Hash Bug)
**Phase:** Core Agent Interaction / Deduplication
**Date Identified:** March 9, 2026
**Severity:** CRITICAL (Silent App Lockout)

### Description
If a driver typed "Navigate to Home Depot", refreshed the page, and typed "Navigate to Home Depot" again in the new session, the backend immediately blocked the request with a "Duplicate request detected" error. The agent would state "I am already processing that request. Please wait." indefinitely.

### Root Cause Analysis
The server-side deduplication mechanism in `core.py` tracked actively processing requests by hashing the message text (`hashlib.md5(message.encode("utf-8"))`). Because the hash ONLY contained the message strings, identical messages from two completely different sessions generated the exact same MD5 collision. Furthermore, the cache lacked a Time-To-Live (TTL). If a request crashed without removing its hash from the active set, that exact string became permanently soft-locked globally across the entire server instance forever.

### Resolution Implemented
1. **Session-Bound Key Hashing:** Altered the hash input structure to explicitly combine both the session ID and the message text (`f"{session_id}:{message}"`). Identical messages from different browser sessions now safely generate entirely distinct MD5 hashes securely.
2. **5-Second TTL Engine:** Intercepted the bare `set()` with a transient `dict` mapping the hash keys against active `time.time()` float timestamps. The deduplicator now aggressively trims any hash execution older than 5.0 seconds automatically, permanently eliminating zombie lockouts.
